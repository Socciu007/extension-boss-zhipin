Schedule + Throttling

Hai cơ chế: chrome.alarms (wake-up) + setTimeout nội bộ (delay giữa reply) + active window check

1. Wake-up loop — chrome.alarms

// src/background/index.ts
chrome.runtime.onInstalled.addListener(() => {
  chrome.alarms.create('auto-reply-tick', { periodInMinutes: 1 })
})

chrome.alarms.onAlarm.addListener((alarm) => {
  if (alarm.name === 'auto-reply-tick') {
    loop.runOnce().catch((e) => storage.recordError(e))
  }
})

Alarm 1 phút/lần. Mỗi tick, runOnce() kiểm tra điều kiện active; nếu pass, mới lặp một reply rồi tự gọi lại qua setTimeout.

2. Active window check

// src/background/scheduler.ts
export function isInActiveWindow(config: AppConfig, now = new Date()): boolean {
  const day = now.getDay()
  const minutes = now.getHours() * 60 + now.getMinutes()
  return config.scheduleWindows.some(w => {
    if (!w.days.includes(day)) return false
    const [sh, sm] = w.start.split(':').map(Number)
    const [eh, em] = w.end.split(':').map(Number)
    return minutes >= sh*60+sm && minutes < eh*60+em
  })
}

Trả về true nếu bất kỳ window nào match. Manual ON/OFF trong popup enabled=false vẫn chặn, dù đang trong active window.

3. Throttle jitter giữa các reply

// src/background/loop.ts (rút gọn)
async function tick(): Promise<void> {
  const { config, stats, enabled } = await storage.getAll()
  if (!enabled) return
  if (stats.date !== today()) await storage.resetDailyStats()
  if (stats.sent >= config.dailyLimit) return
  if (!scheduler.isInActiveWindow(config)) return

  const tabId = await findZhipinTab()
  if (!tabId) return                                   // chưa mở zhipin

  const conv = await pickOneUnreplied(tabId)
  if (!conv) return                                    // inbox rỗng

  const reply = await gemini.generateReply(config, conv)
  await sendReply(tabId, conv.id, reply)

  await storage.markReplied(conv.id)
  await storage.bumpSent()

  // jitter 2-5s
  const wait = randomBetween(config.throttleMinMs, config.throttleMaxMs)
  setTimeout(tick, wait)
}

4. MV3 Service Worker keep-alive

Vấn đề: SW bị Chrome kill sau ~30s idle. setTimeout dài có thể bị nuốt.

Giải pháp: mỗi vòng tick đều trả lời SW bằng cách gọi lại chrome.alarms.create('auto-reply-tick', { delayInMinutes: 0.05 }) (≈3s) khi cần vòng tiếp theo trong tương lai gần. SW chỉ thật sự idle khi thực sự không có việc (không trong active window, hết quota, không có tab zhipin). Khi vào lại active window, alarm 1 phút sẽ đánh thức SW.

Trade-off: SW "ngủ" sâu → tiết kiệm pin; nhưng lúc thoát active window → tự tắt. Cần popup re-enable thì alarm 1 phút là đủ.

5. Race conditions

  • Alarm fires đồng thời nhiều lần do SW wake up trễ: dùng isRunning flag trong storage để chặn tick mới khi tick cũ chưa xong.
  • User đổi config giữa chừng: storage.getAll() đọc mỗi tick → luôn dùng config mới nhất.
  • Tab zhipin bị đóng giữa lúc đang reply: try/catch quanh mọi chrome.tabs.sendMessage; lỗi → storage.recordError() và tiếp tục tick kế tiếp.
