# Auto-Reply on Boss Zhipin — Design

**Date:** 2026-06-09
**Status:** Approved (pending v1 implementation)
**Owner:** BOSS recruiter workflow

## Scope of this design

This spec covers the **end-to-end v1** of the auto-reply feature, including
the service worker, content script, Gemini integration, storage layer, and
the **popup UI** (counter, ON/OFF toggle, API key input, "Test selectors"
button). The popup UI is required to make the feature usable — without it
the user has no way to enable auto-reply or supply an API key.

## Problem

A BOSS recruiter on zhipin.com (the web side, `/web/chat`) needs to acknowledge
new greetings from candidates (`新招呼(197)`) quickly. The current flow is to
manually click each conversation, read the candidate's message, and write a
reply. This is repetitive and time-consuming.

## Goal

When the recruiter is away, the extension auto-replies to unread conversations
in the inbox, with messages generated by Google Gemini based on the candidate's
last message.

## Non-goals (YAGNI)

- Cold outreach / mass messaging new recruiters
- Per-JD message templates (single hardcoded prompt)
- Blacklist/whitelist candidates
- Multi-account support
- AI training or analytics dashboards

## User flow

1. User installs the extension, opens `chrome://extensions/`, loads the unpacked
   `dist/`.
2. User clicks the action icon → popup opens. They paste their Gemini API key
   (Google AI Studio) and click **Save**.
3. User toggles **Auto-reply ON**. The popup shows live counters: sent today,
   errors, whether the scheduler is in active window.
4. User opens `https://www.zhipin.com/web/chat` in a tab. The content script
   boots and waits for commands from the service worker.
5. Service worker's `chrome.alarms` fires every minute. Each tick:
   - Skip if disabled, outside active window, daily limit reached, no zhipin
     tab, or `isRunning` flag set.
   - Ask the content script for the list of conversations with unread badges.
   - Pick the first one that isn't in the replied cache.
   - Open it, read the last candidate message.
   - Call Gemini REST API.
   - Type the reply into the composer and click Send.
   - Mark conversation as replied; bump `sent` counter; schedule the next tick
     with jitter (2-5s).
6. Errors at any step are recorded in storage and shown in the popup.

## Architecture

Three surfaces, one manifest, one service worker.

```
┌──────────────────────────────────────────┐
│  zhipin.com  (chat page)                 │
│  ┌────────────────┐                      │
│  │ Content Script │  scrape DOM          │
│  └────────┬───────┘                      │
│           │ chrome.runtime.sendMessage    │
└───────────┼──────────────────────────────┘
            │
┌───────────▼──────────────────────────────┐
│  Service Worker (background)             │
│  - API key (chrome.storage.local)        │
│  - Gemini REST call                      │
│  - chrome.alarms (1-min tick + jitter)   │
│  - Daily counter & replied cache         │
│  - Race-guard isRunning flag             │
└───────────┬──────────────────────────────┘
            │
┌───────────▼──────────────────────────────┐
│  Popup (counter + ON/OFF + config)       │
└──────────────────────────────────────────┘
```

The service worker is the "brain"; the content script is the "hands and eyes".
This keeps the Gemini API key out of the page world and lets the loop survive
tab navigation.

## Data flow — one reply

1. **Alarm** fires (`periodInMinutes: 1`).
2. **SW `runOnce()`**:
   - `tryAcquireRunLock()` — skip if previous tick still running.
   - `storage.getAll()` — read config + stats.
   - Skip if `!enabled`, no API key, `sent >= dailyLimit`,
     `!isInActiveWindow(config)`, no zhipin tab.
   - `findZhipinTab()` → `chrome.tabs.query`.
   - Send `{ type: 'SCRAPE_UNREAD' }` to content script.
   - Content script returns `Conversation[]` (rows with unread badge).
   - Pick first not in `repliedCache` → `conv`.
   - Send `{ type: 'OPEN_CONV', convId }` → content script clicks the row,
     waits for `[class*="message-list"]` to appear.
   - Send `{ type: 'READ_LAST_MESSAGE', convId }` → content script returns
     the last bubble's text.
   - `gemini.generateReply(config, conv, text)` → returns reply string.
   - Send `{ type: 'SEND_REPLY', convId, text }` → content script types and
     clicks Send (with 3-strategy fallback).
   - `storage.markReplied(conv)`, `storage.bumpSent()`.
   - `setTimeout(jitter(2000, 5000))` → call `runOnce()` again for the next
     reply. The 1-minute alarm is the safety net if SW is killed before
     the timeout fires.
   - `releaseRunLock()` in `finally`.
3. **Errors at any step** → `storage.recordError(msg)`, return early. Conv is
   NOT marked replied → next tick retries.

## State / storage

Single `chrome.storage.local` key: `auto-reply:persisted`.

```ts
type Persisted = {
  config: AppConfig
  conversations: Record<string, Conversation>  // LRU-capped at 2000
  stats: DailyStats                              // reset at midnight local
  enabled: boolean
  isRunning: boolean                             // race-guard
}

type AppConfig = {
  geminiApiKey: string
  model: 'gemini-2.0-flash' | 'gemini-2.5-flash'
  throttleMinMs: number                          // default 2000
  throttleMaxMs: number                          // default 5000
  dailyLimit: number                             // default 200
  scheduleWindows: ScheduleWindow[]              // default weekdays 9-12, 14-18
  systemPrompt: string                           // hardcoded in prompt.ts
}

type ScheduleWindow = {
  days: number[]            // 0=Sun..6=Sat
  start: string             // "HH:MM"
  end: string               // "HH:MM"
}

type Conversation = {
  id: string                // from data-uid OR positional hash
  candidateName: string
  jobTitle: string
  lastSnippet: string
  lastRepliedAt: number     // ms epoch
}

type DailyStats = {
  date: string              // "YYYY-MM-DD" local
  sent: number
  errors: number
  lastErrorMsg: string
}
```

`storage.getAll()` defensively merges with `DEFAULT_PERSISTED` so new fields
in future builds don't crash old data.

## Schedule + throttling

Two-layer wake-up:

1. **Primary**: `chrome.alarms.create('auto-reply-tick', { periodInMinutes: 1 })`
   — fires every minute. This is the **safety net** that wakes the SW after
   Chrome kills it for inactivity.
2. **Secondary**: `setTimeout(jitter(2000, 5000))` inside `runOnce()` to chain
   the next reply with human-like delay. Chrome MV3 service workers are
   allowed to be killed after ~30s idle, so a 2-5s `setTimeout` usually
   completes; if it doesn't, the 1-minute alarm picks up the loop.

**Why not a 3-second alarm?** Chrome enforces a 30-second minimum
`delayInMinutes` in production. `setTimeout` is the right tool for
sub-minute jitter.

- `isInActiveWindow(config)` — pure check: now's day-of-week and HH:MM
  intersect any window.
- `jitter(min, max)` — uniform random integer.
- Manual `enabled=false` overrides schedule: loop returns immediately.

**SW startup recovery**: on every SW cold start (`chrome.runtime.onStartup`
and the alarm re-creation callback in `background/index.ts`), call
`storage.releaseRunLock()`. If a previous tick was killed mid-loop, the
`isRunning` flag would otherwise stay `true` forever.

## DOM scraping strategy

**Goal**: survive BOSS rolling new builds.

**Priority order** for selectors:
1. `data-*` attributes (most stable if BOSS sets them).
2. Structural / positional (e.g., `aside ul > li`).
3. Text content match.
4. Hash class (e.g., `[class*="badge"]`) — last resort, marked as
   fragile in comments.

Each logical element has an **array** of fallback selectors — `dom.$` tries
them in order, returns the first match.

For **typeAndSend** (composer), three strategies tried in order:
1. `textContent = ''; append(textNode); dispatchEvent('input')` — works for
   most React contenteditables.
2. `document.execCommand('insertText', false, text)` — older fallback.
3. `dispatchEvent(new ClipboardEvent('paste', { clipboardData }))` — for
   ProseMirror-style composers.

A **selector test** action is exposed via `chrome.runtime.onMessage` so the
popup can show matched counts per selector — used to verify after BOSS
updates their site.

## Gemini integration

- Endpoint:
  `https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent?key={key}`
- Request body:
  - `systemInstruction.parts[0].text` = hardcoded Chinese recruiter persona
    (see `src/content/features/auto-reply/prompt.ts`).
  - `contents[0].parts[0].text` = context block with candidate name, job
    title, last message, and "Hãy trả lời:".
  - `generationConfig`: temperature 0.6, maxOutputTokens 256, topP 0.9.
  - `safetySettings`: all four categories set to `BLOCK_NONE` (recruiter
    business language should never trigger; safety filter could otherwise
    return empty `candidates`).
- If the response has no `candidates[0].content.parts[0].text` (safety
  filter blocked), fall back to `FALLBACK_REPLY` so the candidate gets
  *something* rather than being silently dropped.

## Error handling

| Layer | Failure mode | Handling |
|---|---|---|
| DOM scrape | selector mismatch, timeout | empty array returned → loop skips tick |
| Open conv / read msg | click no-op, pane not render | `recordError`, skip conv this tick |
| Gemini API | 4xx/5xx, safety, rate limit, network | `recordError`, conv NOT marked replied → retry next tick |
| typeAndSend | composer gone, 3 strategies fail, send disabled | `recordError`, conv NOT marked replied → retry |
| Service worker | Chrome kills mid-loop | `try/finally` + `releaseRunLock`; on SW restart (`onStartup` + alarm re-create) call `releaseRunLock` to clear any stuck flag; 1-min alarm retries |

**Retry policy**: failed conversations are retried indefinitely on the next
tick (no skip-after-N). This is simple and good enough for v1 — the
recruiter can manually reply or open the conv to "unstick" it.

**Circuit breaker**: not in v1. If needed later, add: 3 consecutive
identical errors in 10 min → pause loop 10 min.

## Manifest changes

- Add `alarms` permission (already present in user's draft).
- Remove `host_permissions: 'https://*/*'`. Replace with:
  - `https://*.zhipin.com/*` (content script)
  - `https://generativelanguage.googleapis.com/*` (Gemini API from SW)
- Add `background.service_worker: 'src/background/index.ts'` with
  `type: 'module'`.
- `content_scripts[0]`:
  - `js: ['src/content/features/auto-reply/index.ts']`
  - `matches: ['https://*.zhipin.com/web/chat/*']`
  - `run_at: 'document_idle'`
- `src/content/features/index.ts` (the oocl.com stub) is **deleted**.
